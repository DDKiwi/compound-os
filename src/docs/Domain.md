# Domain

## Purpose

This document defines the business language of Compound OS.

It establishes the concepts, relationships and rules that make up the investment domain.

Every feature, engine and user interface should use the terminology defined here.

If a new concept cannot be described within this document, it should be questioned before implementation.

---

# Domain Philosophy

Compound OS is an operating system for long-term investors.

Its purpose is not to predict markets.

Its purpose is to help investors make rational, repeatable and informed decisions.

The domain should remain independent of:

* frameworks
* APIs
* databases
* user interface
* external services

Business concepts must never be defined by implementation details.

---

# Core Concepts

## Investor

The person using Compound OS.

The investor defines an Investment Policy and owns one or more Portfolios.

---

## Portfolio

A portfolio represents a collection of assets owned by an investor.

A portfolio has:

* holdings
* allocation
* performance
* income
* risk
* goals

A portfolio is evaluated against an Investment Policy.

---

## Asset

An asset represents something that can be owned.

Examples include:

* Stocks
* ETFs
* Funds
* Bonds
* Cash
* REITs

Future asset types should integrate naturally without changing the overall domain model.

---

## Holding

A holding represents ownership of an asset.

A holding contains information such as:

* quantity
* average cost
* current value
* dividend history
* allocation

Holdings belong to a Portfolio.

---

## Investment Policy

The Investment Policy is the central object of Compound OS.

Every engine should evaluate the portfolio against the policy.

An Investment Policy may define:

* investment philosophy
* target allocation
* maximum position size
* sector exposure
* dividend strategy
* cash allocation
* rebalancing thresholds
* valuation preferences
* buying rules
* selling rules
* risk tolerance

The policy should express intent rather than implementation.

---

## Goal

A goal describes what the investor wants to achieve.

Examples include:

* Financial independence
* Passive income
* Capital growth
* Wealth preservation

Goals influence recommendations but do not override the Investment Policy.

---

## Recommendation

A recommendation is a suggested action generated by the system.

Recommendations should always explain:

* what
* why
* confidence
* expected impact

Compound OS assists the investor.

It never makes decisions on the investor's behalf.

---

## Signal

A signal is an observation.

Signals describe facts.

Examples:

* Dividend increased
* Position exceeds allocation
* Valuation below target
* Portfolio concentration increased

Signals do not recommend actions.

---

## Rule

A rule evaluates one aspect of a portfolio.

Rules return objective outcomes.

Examples:

* Pass
* Warning
* Fail

Rules should remain deterministic.

---

# Domain Engines

The domain is composed of specialized engines.

Each engine has a single responsibility.

Examples:

* Portfolio Engine
* Dividend Engine
* Risk Engine
* Dashboard Engine
* Recommendation Engine

Engines communicate through domain objects rather than direct dependencies.

---

# Principles

Business rules should be:

* deterministic
* testable
* composable
* framework independent

The same input should always produce the same result.

---

# Future Evolution

The domain should support future expansion without redesign.

Examples include:

* multiple portfolios
* additional asset classes
* tax analysis
* retirement planning
* scenario simulations
* AI-assisted recommendations

New functionality should extend the domain—not replace it.

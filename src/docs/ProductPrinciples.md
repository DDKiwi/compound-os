# Product Principles

## Purpose

These principles define what Compound OS should become.

Every new feature, workflow and design decision should support these principles.

If a proposal conflicts with these principles, it should be reconsidered before implementation.

---

# Compound OS exists to improve decision making.

The application does not exist to predict the market.

It exists to help investors make rational, consistent and repeatable decisions.

---

# The investment policy is the source of truth.

Every analysis, recommendation and evaluation begins with the user's Investment Policy.

Compound OS should never replace the user's philosophy.

It should help the user follow it.

---

# Explain before recommending.

Every recommendation should answer:

* What happened?
* Why does it matter?
* Why is this recommended?

Recommendations without explanations should not exist.

---

# Data before decoration.

Information is the product.

Visual design exists to improve understanding—not to impress.

---

# Calm over urgency.

The application should reduce emotional investing.

Avoid unnecessary alerts, flashing indicators and attention-seeking UI.

Urgency should only be communicated when objectively justified.

---

# Consistency over complexity.

Simple, predictable behavior is preferred over feature richness.

A smaller set of well-designed features is better than many disconnected ones.

---

# The investor remains in control.

Compound OS informs.

The investor decides.

The system should never perform investment actions automatically.

---

# Build for the long term.

Design decisions should support years of evolution.

Avoid solutions that only solve today's problem.
